## Array
- 메모리 상 연속한 구간 잡아야 해서 할당에 제약
- Cache hit rate 높음

## LinkedList
- k번째 원소 확인/변경: O(k)
- 메모리 상 연속하지 않아 할당이 쉬움
- Cache hit rate 낮음

**erase(it)**: 현재 iterator 제거 후 다음 iterator 반환
    - it = erase(it) 형태로 써야 안전하다 !!
    - erase(prev(it)): 전 원소 제거(backspace)
    - erase(next(it)): 다음 원소 제거
**insert(it, value)**: 현재 iterator 왼쪽에 value 삽입

## Array와 LinkedList
둘 다 선형 자료구조 이다. 둘의 차이를 알아두자.
|                          | 배열  | 연결리스트 |
| k번째 원소 접근           | O(1)  | O(k)      |
| 임의 위치에 원소 추가/제거 | O(N)  | O(1)      |
| 메모리 상의 배치          | 연속  | 불연속     |
| Overhead                 | -    | O(N)       |

Overhead는 배열은 굳이 따지면 길이를 저장하겠지만
연결리스트는 전 또는 후 pointer를 계속 저장해야 해서
overhead가 발생한다.

## Map vs Unordered Map

1. 내부 구현 차이
- Map: 레드-블랙 트리(균형 이진 탐색 트리)
    - 범위 기반 탐색이 필요하다면 사용
- UnorderedMap: 해시 테이블
    - 빠른 조회가 필요할 때 사용

2. 정렬
- Map: 키 기준 자동 정렬
- UnorderedMap: 정렬 x

3. 시간 복잡도
Search, Insert, Delete에 대해
- Map: O(log N)
- UnorderedMap: O(1) ~ O(N)

#### 형변환
- string to int
    - `stoi(s)`: 문자열을 int로 변환. `<string>` 헤더 필요.
    ```cpp
    int num = stoi("123");   // 123
    ```
    - 관련 함수: `stol`(long), `stoll`(long long), `stof`(float), `stod`(double)
    - 주의: 변환 범위를 초과하면 예외 발생. 큰 수는 `stoll` 사용.
    - sstream 방식도 가능:
    ```cpp
    string s = "123";
    stringstream ss(s);
    int num; ss >> num;
    ```
- int to string
    - `to_string(num)`: 숫자를 문자열로 변환. `<string>` 헤더 필요.
    ```cpp
    string str = to_string(42);   // "42"
    ```
    - **독립 함수**이므로 `num.to_string()` 형태로 쓸 수 없다 (멤버 함수 아님).


#### 문자열 비교 팁
- **같은 길이의 숫자 문자열**은 사전순(lexicographic) 비교 = 크기 비교이다.
    ```cpp
    "621" > "612"  // true, 숫자로도 621 > 612
    ```
- 따라서 `stoi`로 변환할 필요 없이 문자열 그대로 비교하면 된다.
- 오버플로우 걱정도 없고 코드도 간결해진다.
- 활용 예: 커스텀 정렬에서 두 조합을 만들어 비교할 때 (`case1 > case2`)

#### 기타
- 큰 숫자를 다룰 때는 string으로 다루는 것도 방법

앞 뒤로 넣고 빼는 자료구조.
대표적으로 push_front(), push_back(), pop_front(), pop_back() 여기에 추가로 insert도 존재

**Dequq와 Vector의 차이?**

Vector는 연속적인 메모리 공간에 저장

- 장점: 개별 원소 접근 빠름, 끝 삽입/제거 빠름
- 단점: 처음~중간 삽입, 제거 비용 큼, 확장 비용 큼
    - 메모리 상의 시퀀스를 유지하려는 특성
    - 공간이 부족하면 해당 길이를 수용할 수 있는 새로운 메모리가 필요

Deque는 vector와 유사하나 다름.
임의 접근이 가능하다(다만 벡터보다 느림)

- 장점: 앞, 뒤 삽입/제거 빠름
- 단점: 연속적인 메모리 공간이 아니므로 원소들간 포인터 연산이 불가능

List == Linked List

- 장점: 어느 위치에나 삽입/제거 빠름
- 단점: index를 통한 임의 접근 불가

## [BOJ 1074] Z
- **주요 개념**: 재귀(Recursion), 분할 정복(Divide and Conquer)
- **배운 점**: 
    1. **상태 변화**: 재귀 호출 시 문제의 규모가 어떻게 줄어드는지(N -> N-1) 정확히 파악해야 함.
    2. **비트 연산**: 2의 거듭제곱(2^n)은 pow(2, n)보다 (1 << n)이 더 정확하고 빠름.
    3. **출력문 주의**: 디버깅을 위해 넣은 cout은 제출 전 반드시 제거해야 '출력 형식이 잘못되었습니다'를 방지할 수 있음.
    4. **시간 복잡도**: 4분면 중 한 곳만 골라서 들어가는 방식은 O(N)의 깊이를 가지므로 큰 범위도 효율적으로 해결 가능.

## 문제별 메서드/문법 기록

### [BOJ 17478] 재귀함수가뭔가요
- 헷갈린 문법:
```cpp
string underbar(x * 4, '_');
```
- 의미:
  - 길이가 `x * 4`인 문자열을 만들고, 모든 문자를 `'_'`로 채운다.
  - 즉, `"____"`를 `x`번 반복한 것과 같은 들여쓰기 효과를 만든다.
- 왜 필요했는가:
  - 재귀 깊이에 따라 같은 패턴의 접두 문자열(들여쓰기)을 빠르게 생성해야 하기 때문.
- 비교 메모:
  - `underbar * x` 형태는 C++ `string`에서 지원되지 않는다.
  - C++은 파이썬처럼 문자열 곱셈 문법이 없어서 위 생성자 방식으로 처리한다.

### 문자열 생성자 기본형 메모
- 자주 쓰는 형태:
```cpp
string s(count, ch);
```
- 예시:
```cpp
string stars(5, '*');   // "*****"
string zeros(3, '0');   // "000"
```

### getline() 사용법
- `cin >>`은 공백에서 끊기므로, 공백 포함 한 줄 입력은 `getline`을 사용한다.
```cpp
string a;
getline(cin, a);
```
- `cin >>`과 섞어 쓸 때는 버퍼에 남은 `\n`에 주의 (`cin.ignore()` 필요할 수 있음)

### string::find()
- 특정 부분 문자열을 찾는 메서드. 찾은 문자열의 첫 인덱스를 반환하고, 못 찾으면 `string::npos` 반환.
```cpp
str.find(subStr, pos); // pos 위치부터 subStr을 탐색
```
- 겹치지 않게 찾기: `pos += subStr.size()`
- 겹쳐서 찾기: `pos += 1`

### string::replace()
- 문자열의 특정 위치부터 지정 길이만큼을 다른 문자열로 치환한다.
```cpp
s.replace(pos, length, newStr);
// pos: 시작 위치, length: 치환할 길이, newStr: 대체 문자열
```
- `find()`와 조합하면 특정 패턴을 일괄 치환하는 패턴으로 활용 가능:
```cpp
while ((pos = s.find(pattern, pos)) != string::npos)
{
    s.replace(pos, pattern.size(), "#");
    pos += 1;
}
```

### string::substr()
- 문자열의 일부를 추출한다. 두 번째 인자는 **길이**이다 (끝 인덱스가 아님).
```cpp
s.substr(pos, len);   // pos부터 len개 문자 추출
s.substr(pos);         // pos부터 끝까지 (len 생략 가능)
```

### string::rfind()
- `find()`의 역방향 버전. 뒤에서부터 탐색하여 마지막으로 등장하는 위치를 반환한다.
```cpp
s.rfind(pattern);      // 뒤에서부터 pattern 탐색
```
- suffix 매칭에 유용: `s.rfind(suffix) + suffix.size() == s.size()`로 끝부분 일치를 확인할 수 있다.

### char* → string 변환
- 대입 또는 생성자로 바로 변환 가능:
```cpp
char* cstr = "hello";
string s1 = cstr;    // 대입
string s2(cstr);     // 생성자
```

## 260327 프로그래머스 정렬 문제 정리

### 문제: 프로그래머스 H-Index (정렬)
- 핵심 개념: 정렬 후 선형 탐색, H-Index 정의 이해
- H-Index 정의: 논문 n편 중 h번 이상 인용된 논문이 h편 이상이고, 나머지 논문이 h번 이하 인용된 값 중 최댓값
- 풀이 흐름: 오름차순 정렬 → i번째부터 뒤에 `n-i`편 존재 → `n-i <= c[i]`인 첫 지점에서 `answer = min(c[i], n-i)`
- 내 실수/막힌 지점: **문제 정의를 잘못 이해**하여 불필요한 `continue` 조건을 추가함 (이전 원소가 h 이상이면 건너뛰는 로직). 이로 인해 `[1,1,1]` 같은 동일값 케이스에서 오답 발생
- 교정 포인트: 복잡한 예외 조건을 추가하기 전에 **문제 정의를 정확히 재확인**하는 습관. 정렬 후 `n-i <= c[i]`인 첫 지점이 곧 최적 h라는 단순한 논리로 충분했음
- 사용한 메서드/자료구조/알고리즘: `sort()`, 선형 탐색, `min()`
- 시간복잡도/공간복잡도: O(N log N), O(1)
- 다음에 재사용할 패턴: "오름차순 정렬 후 `n-i`와 `c[i]` 비교" — H-Index 류 문제의 정석 접근

### 문제: 프로그래머스 다리를 지나는 트럭 (스택/큐)
- 핵심 개념: 큐로 다리 위 트럭 관리 + 시간 점프 최적화
- 풀이 흐름: queue에 (무게, 진입시간) 저장 → 매 시점마다 탈출 트럭 pop → 트럭 추가 시도 → 못 올리면 선두 탈출 시점으로 시간 점프
- 내 실수/막힌 지점:
    1. 초기 풀이에서 시간을 1씩 증가 → O(N * bridge_length) 비효율
    2. 시간 점프 적용 시, pop한 트럭의 시간으로 점프 → 시간이 뒤로 가는 버그
- 교정 포인트:
    - 시간 점프는 반드시 **현재 큐의 front** 기준: `t = q.front().y + bridge_length`
    - 탈출 처리를 while로 분리하면 여러 트럭 동시 탈출도 처리 가능
    - 마지막 트럭 탈출 시점: `q.back().y + bridge_length`로 한 줄 계산
- 구조 패턴 (깔끔한 버전):
    ```cpp
    while (i < size) {
        while (!q.empty() && 탈출조건) pop;  // 1. 탈출
        if (올릴수있으면) push, t++;          // 2. 추가
        else t = front 탈출시점;              // 3. 점프
    }
    return q.back().y + bridge_length;
    ```
- 사용한 메서드/자료구조/알고리즘: `queue<pair>`, 시간 점프 시뮬레이션
- 시간복잡도/공간복잡도: O(N), O(N)
- 다음에 재사용할 패턴: "탈출 while + 추가 if + 점프 else" 다리/컨베이어 시뮬레이션 템플릿

### 프로그래머스 Lv.2 — 타겟 넘버
- 핵심 개념: DFS 완전탐색 — 각 숫자에 +/- 두 선택지 → 이진 트리 탐색
- 풀이 흐름: dfs(i, 부분합) → i번째 숫자를 더하거나 빼고 재귀 → 끝까지 가면 target 비교
- 시간복잡도: O(2^N), N ≤ 20이므로 2^20 = ~100만 → 충분
- 사용한 메서드/자료구조/알고리즘: DFS 재귀, 완전탐색
- 다음에 재사용할 패턴: "각 원소에 2가지 선택" → DFS로 모든 경우 탐색 (N ≤ 20이면 O(2^N) OK)

### 프로그래머스 Lv.3 — 네트워크
- 핵심 개념: BFS + 연결 요소(Connected Component) 개수 세기
- 풀이 흐름: 미방문 노드 발견 → BFS로 연결된 노드 전부 방문 처리 → answer++ 반복
- 시간복잡도: O(N²) — 인접 행렬 순회
- 주의: 전역 visited 배열은 프로그래머스에서 테스트케이스 간 초기화 안 될 수 있음 → solution 시작 시 memset 필요
- 사용한 메서드/자료구조/알고리즘: BFS(queue), visited 배열, 인접 행렬
- 다음에 재사용할 패턴: "연결 요소 개수" = 미방문 노드마다 BFS/DFS 돌린 횟수
